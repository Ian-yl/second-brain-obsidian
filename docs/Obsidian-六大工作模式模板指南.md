# Obsidian 六大工作模式目录结构指南

> 这版只解决 **目录结构**，不进入具体笔记内容模板。
> 核心思路：保留卡帕西式知识四层 `Inputs / Process / Outputs / Feedback`，但不按固定行业建 24 套系统；用 6 大工作模式承接所有行业、角色和场景。

## 1. 为什么不按行业建目录

行业不是稳定边界。

一方面，行业远远不止 24 个：医疗、法律、教育、制造、咨询、游戏、电商、供应链、政务、公益、能源、农业、保险、媒体、AI、机器人、建筑、餐饮、旅游、体育、心理咨询、客户成功、自由职业、家庭管理等都可以继续展开。

另一方面，一个真实用户往往不是单一行业身份：

- 创业者同时做产品、销售、内容、融资、招聘。
- 医生可能同时做临床、科研、教学、科普。
- 咨询顾问同时做研究、客户管理、项目交付。
- 产品经理同时做用户研究、项目推进、需求文档、上线复盘。

所以目录不应该问：

```text
你属于哪个行业？
```

而应该问：

```text
你现在主要在做哪类工作？
```

行业适合做 **映射层**，不适合作为一级目录。

## 2. 两层设计

### 2.1 第一层：知识四层

所有知识都按流转阶段进入系统：

| 层级 | 含义 | 放什么 |
|---|---|---|
| `Inputs` | 输入 | 灵感、摘录、访谈、资料、反馈、观察、信源 |
| `Process` | 加工 | 分析、草稿、方案、决策、实验、推理、会议 |
| `Outputs` | 输出 | 文章、报告、PRD、交付物、发布记录、作品 |
| `Feedback` | 反馈 | 复盘、评论、指标回看、事故、改进项、经验教训 |

这四层回答的是：**知识现在处在什么生命周期？**

### 2.2 第二层：六大工作模式

六大工作模式回答的是：**这条知识服务于哪类工作？**

目录面向人，默认使用中文名称。英文只作为自动化、脚本、映射表里的稳定 ID。

| 中文目录名 | 英文 ID | 适合管理什么 |
|---|---|---|
| `研究分析` | `research-analysis` | 资料、问题、证据、分析、结论 |
| `项目交付` | `project-delivery` | 目标、项目、需求、决策、里程碑、交付 |
| `客户个案` | `case-management` | 客户、病例、案件、候选人、学生、房源 |
| `内容生产` | `content-production` | 选题、素材、草稿、发布、内容反馈 |
| `流程运营` | `operations` | SOP、清单、异常、事故、审计、改进 |
| `学习成长` | `learning-growth` | 课程、概念、练习、日记、习惯、回顾 |

### 2.3 两层关系

四层是纵向流转，六大模式是横向工作场景。

```text
同一条笔记 = 一个知识层级 + 一个工作模式 + 一个可选行业/角色
```

例如：

```text
用户访谈记录
→ Inputs
→ 研究分析
→ domain: product
```

```text
上线复盘
→ Feedback
→ 项目交付
→ domain: software / product
```

```text
病例学习
→ Process
→ 客户个案
→ domain: healthcare
```

## 3. 一个通用 Vault 结构

只保留一个通用结构：

```text
Obsidian-Vault/
├── 00-Inbox/
├── 10-Inputs/
├── 20-Process/
├── 30-Outputs/
├── 40-Feedback/
├── 50-MOCs/
├── 60-Domains/
├── 70-Assets/
├── 90-Archive/
└── _System/
    ├── Templates/
    ├── Dashboards/
    └── Attachments/
```

### 3.1 目录说明

| 目录 | 用途 |
|---|---|
| `00-Inbox/` | 临时收集箱，未判断归属的内容先放这里 |
| `10-Inputs/` | 外部输入和原始材料 |
| `20-Process/` | 正在加工、思考、推进的内容 |
| `30-Outputs/` | 已形成的产出、作品、交付物 |
| `40-Feedback/` | 结果反馈、复盘、经验、改进项 |
| `50-MOCs/` | 地图页，按主题/项目/工作模式建立入口 |
| `60-Domains/` | 行业、角色、领域映射页 |
| `70-Assets/` | 图片、PDF、附件、素材等资源 |
| `90-Archive/` | 完结、废弃、历史内容 |
| `_System/` | 模板、看板、附件配置等系统文件 |

这个结构的关键是：**四层知识流作为主目录，六大工作模式不再各自建一套根目录。**

否则会出现两种分类互相打架：

```text
研究分析/Inputs/
项目交付/Inputs/
客户个案/Inputs/
...
```

这种结构后期会膨胀。更好的做法是主目录只按四层走，工作模式用固定子目录和 MOC 承接。

## 4. 六大工作模式怎么放

六大工作模式不作为六个一级根目录，而是在每个知识层下作为**按需创建的中文子目录**，并通过 MOC 做导航入口。

### 4.1 建库默认只创建一级目录

```text
Obsidian-Vault/
├── 00-Inbox/
├── 10-Inputs/
├── 20-Process/
├── 30-Outputs/
├── 40-Feedback/
├── 50-MOCs/
├── 60-Domains/
├── 70-Assets/
├── 90-Archive/
└── _System/
    ├── Templates/
    ├── Dashboards/
    └── Attachments/
```

建库时不要一次性生成 `4 层 × 6 模式 = 24` 个二级目录。空目录太多会让用户误以为每个格子都必须填，也会制造选择压力。

### 4.2 二级模式目录按需创建

当用户第一次创建某类笔记时，再自动创建对应二级目录。

示例：

```text
用户导入一篇行业报告
→ 10-Inputs/研究分析/

用户开始写一个 PRD
→ 20-Process/项目交付/

用户保存一篇已发布文章
→ 30-Outputs/内容生产/

用户做一次事故复盘
→ 40-Feedback/流程运营/
```

二级目录只使用这 6 个中文名称：

```text
研究分析
项目交付
客户个案
内容生产
流程运营
学习成长
```

英文 ID 不进入默认目录名，只在自动化规则里使用。

## 5. 行业如何兼容

行业不进一级目录，放在 `60-Domains/` 里做映射页。

```text
60-Domains/
├── product.md
├── software.md
├── healthcare.md
├── legal.md
├── education.md
├── finance.md
├── consulting.md
├── manufacturing.md
├── ecommerce.md
└── custom/
```

每个行业页只回答三件事：

```text
这个行业常用哪些工作模式？
典型内容分别落在哪个知识层？
推荐从哪些 MOC 进入？
```

### 5.2 新行业扩展规则

新增行业时，不新增一套目录，只新增一张 domain 映射页。

```text
60-Domains/<domain>.md
```

映射页包含：

```text
标题：<行业/角色名>

常用工作模式：
- 主模式：
- 辅助模式：

常见内容落位：
- Inputs：
- Process：
- Outputs：
- Feedback：

推荐入口 MOC：
- [[MOC - <工作模式>]]
- [[MOC - <项目/主题>]]
```

## 6. 自动判断工作模式

自动提炼时，不让用户手动判断模式。系统按 **对象 + 动作 + 产出** 三个线索判断。

```text
对象：这条内容主要管理什么？
动作：用户正在做什么？
产出：最后要形成什么结果？
```

### 6.1 快速判定表

| 主要线索 | 判为 |
|---|---|
| 资料、文献、证据、调研、数据、结论、洞察 | 研究分析 |
| 项目、需求、里程碑、方案、决策、交付、发布 | 项目交付 |
| 客户、病例、案件、候选人、学生、房源、联系人 | 客户个案 |
| 选题、素材、草稿、文章、视频、课程、发布 | 内容生产 |
| SOP、清单、告警、事故、审计、异常、改进 | 流程运营 |
| 课程、概念、练习、日记、习惯、复习、反思 | 学习成长 |

### 6.2 冲突时按当前目的判断

同一类内容可能进入不同模式，关键看当前目的。

```text
用户访谈
- 为了发现需求 → 研究分析
- 为了推进项目 → 项目交付
- 为了维护客户关系 → 客户个案
- 为了写案例文章 → 内容生产
```

如果仍然无法判断，先放进 `00-Inbox/`，稍后再归类。

### 6.3 自动创建目录规则

确定知识层和工作模式后，如果目标二级目录不存在，就自动创建。

```text
目标路径 = <知识层目录>/<中文工作模式>/
```

示例：

```text
行业报告 → 10-Inputs/研究分析/
需求方案 → 20-Process/项目交付/
发布文章 → 30-Outputs/内容生产/
事故复盘 → 40-Feedback/流程运营/
```

## 7. 用户选择逻辑

用户不需要从几十个行业里选择。

### 7.1 默认提问

```text
你主要想用 Obsidian 管什么？
可以直接说：项目、研究、客户、病例、案件、内容、流程、学习、复盘，或者说你的职业/场景。
```

### 7.2 系统自动判断

```text
用户说：我是产品经理，想管需求和用户反馈。

系统推荐：
- 主工作模式：项目交付
- 辅助工作模式：研究分析
- 知识层落位：
  - 用户反馈 → 10-Inputs/研究分析/
  - 需求分析 → 20-Process/项目交付/
  - PRD/路线图 → 30-Outputs/项目交付/
  - 上线复盘 → 40-Feedback/项目交付/
- domain：product
```

### 7.3 用户只看到少量选择

前台最多展示：

```text
推荐方案：产品经理工作区
包含：项目交付 + 研究分析

你可以：
1. 直接使用
2. 调整工作模式
3. 换一个场景描述
```

不要展示几十个行业按钮。

## 9. 目录使用原则

### 9.1 先判断知识层，再判断工作模式

放笔记时按这个顺序：

1. 它是输入、加工、输出，还是反馈？
2. 它属于哪类工作模式？
3. 它属于哪个行业/角色/项目？

例如：

```text
客户访谈原始记录
→ Inputs
→ 客户个案 或 研究分析
→ domain: sales / product / consulting
```

### 9.2 行业不决定目录

不要因为是“医疗”就放进一个巨大的 `Healthcare/` 文件夹。

医疗里的内容也有：

- Inputs：指南、检查结果、病史资料。
- Process：病例分析、鉴别诊断、教学准备。
- Outputs：处置计划、课件、科普文章。
- Feedback：复诊记录、病例复盘、学习反思。

行业只是解释上下文，不应该吞掉知识流转结构。

### 9.3 MOC 负责导航

目录负责存放，MOC 负责入口。

建议至少建立：

```text
50-MOCs/
├── MOC - 研究分析.md
├── MOC - 项目交付.md
├── MOC - 客户个案.md
├── MOC - 内容生产.md
├── MOC - 流程运营.md
└── MOC - 学习成长.md
```

如果某个行业常用，再加：

```text
60-Domains/product.md
60-Domains/healthcare.md
60-Domains/legal.md
```

### 9.4 不急着细分

初始目录不要过细。建库时只创建一级目录；二级工作模式目录等第一次用到时再创建。

优先级：

```text
先有一级目录
再按需创建模式目录
最后才考虑更细分类
```

建库默认只创建一级目录：

```text

├── 20-Process/
├── 30-Outputs/
├── 40-Feedback/
├── 50-MOCs/
```
