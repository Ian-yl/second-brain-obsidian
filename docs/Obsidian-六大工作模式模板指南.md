# Obsidian Vault 目录规则（LLM 版）

## 0. 任务边界

本规则只用于 **目录结构与文件落位**。

可以做：

- 创建默认一级目录。
- 按需创建二级工作模式目录。
- 判断笔记应该放入哪个知识层。
- 判断笔记属于哪种工作模式。
- 创建少量索引文件。
- 创建行业/角色映射页。

不可以做：

- 不要生成具体笔记内容模板。
- 不要一次性生成 24 个二级目录。
- 不要按行业创建一级目录。
- 不要让用户从几十个行业中手动选择。
- 不要把英文工作模式名作为默认目录名。
- 不要把 `60-Domains/` 当成主要内容存放区。

---

## 1. 默认一级目录

新建 Vault 时，只创建以下一级目录和入口文件：

```text
Obsidian-Vault/
├── 00-Home.md
├── 00-Inbox/
├── 10-Inputs/
├── 20-Process/
├── 30-Outputs/
├── 40-Feedback/
├── 50-MOCs/
├── 60-Domains/
├── 70-Assets/
├── 90-Archive/
└── _System/
    ├── Templates/
    ├── Dashboards/
    └── Attachments/
```

目录用途：

| 路径 | 用途 |
|---|---|
| `00-Home.md` | 总入口索引 |
| `00-Inbox/` | 未归类内容 |
| `10-Inputs/` | 输入材料 |
| `20-Process/` | 加工中内容 |
| `30-Outputs/` | 已产出内容 |
| `40-Feedback/` | 反馈、复盘、改进 |
| `50-MOCs/` | 工作模式索引、主题索引 |
| `60-Domains/` | 行业/角色映射页 |
| `70-Assets/` | 附件、图片、PDF、素材 |
| `90-Archive/` | 归档内容 |
| `_System/` | 系统文件、模板、看板、附件配置 |

---

## 2. 知识四层

所有笔记必须先判断知识层。

| 知识层 | 目录 | 放什么 |
|---|---|---|
| Inputs | `10-Inputs/` | 灵感、摘录、资料、访谈、观察、信源、原始反馈 |
| Process | `20-Process/` | 分析、草稿、方案、决策、实验、推理、会议 |
| Outputs | `30-Outputs/` | 文章、报告、PRD、交付物、发布记录、作品 |
| Feedback | `40-Feedback/` | 复盘、评论、指标回看、事故、改进项、经验教训 |

不能判断时，放入：

```text
00-Inbox/
```

---

## 3. 六大工作模式

工作模式作为二级目录，按需创建。

默认中文目录名：

```text
研究分析
项目交付
客户个案
内容生产
流程运营
学习成长
```

英文 ID 只用于自动化，不作为默认目录名。

| 中文目录名 | 英文 ID | 使用场景 |
|---|---|---|
| 研究分析 | `research-analysis` | 资料、文献、证据、调研、数据、结论、洞察 |
| 项目交付 | `project-delivery` | 项目、需求、里程碑、方案、决策、交付、发布 |
| 客户个案 | `case-management` | 客户、病例、案件、候选人、学生、房源、联系人 |
| 内容生产 | `content-production` | 选题、素材、草稿、文章、视频、课程、发布 |
| 流程运营 | `operations` | SOP、清单、告警、事故、审计、异常、改进 |
| 学习成长 | `learning-growth` | 课程、概念、练习、日记、习惯、复习、反思 |

---

## 4. 二级目录创建规则

不要预创建所有二级目录。

当某条笔记第一次需要某个模式目录时，才创建：

```text
<知识层目录>/<中文工作模式>/
```

示范：

```text
行业报告
→ 10-Inputs/研究分析/

需求方案
→ 20-Process/项目交付/

已发布文章
→ 30-Outputs/内容生产/

事故复盘
→ 40-Feedback/流程运营/

病例记录
→ 20-Process/客户个案/

课程笔记
→ 20-Process/学习成长/
```

禁止：

```text
10-Inputs/
├── 研究分析/
├── 项目交付/
├── 客户个案/
├── 内容生产/
├── 流程运营/
└── 学习成长/

20-Process/
├── 研究分析/
...
```

除非这些目录已经被实际内容用到，否则不要批量创建。

## 5. 工作模式自动判断

按三个线索判断：

```text
对象：这条内容主要管理什么？
动作：用户正在做什么？
产出：最后形成什么结果？
```

快速规则：

| 线索 | 判为 |
|---|---|
| 资料、文献、证据、调研、数据、结论、洞察 | 研究分析 |
| 项目、需求、里程碑、方案、决策、交付、发布 | 项目交付 |
| 客户、病例、案件、候选人、学生、房源、联系人 | 客户个案 |
| 选题、素材、草稿、文章、视频、课程、发布 | 内容生产 |
| SOP、清单、告警、事故、审计、异常、改进 | 流程运营 |
| 课程、概念、练习、日记、习惯、复习、反思 | 学习成长 |

冲突时，按当前目的判断。

示范：

```text
用户访谈
- 为了发现需求 → 10-Inputs/研究分析/
- 为了推进项目 → 20-Process/项目交付/
- 为了维护客户关系 → 20-Process/客户个案/
- 为了写案例文章 → 10-Inputs/内容生产/
```

无法判断时：

```text
00-Inbox/
```

## 6. 行业/角色处理规则

行业不作为一级目录。

行业只放在：

```text
60-Domains/
```

行业页按需创建，不全量预建。

示范：

```text
60-Domains/product.md
60-Domains/healthcare.md
60-Domains/legal.md
60-Domains/consulting.md
```

行业页只写映射，不存主内容。

行业页应包含：

```text
标题：<行业/角色名>

主工作模式：
- 

辅助工作模式：
- 

常见落位：
- Inputs：
- Process：
- Outputs：
- Feedback：

入口索引：
- 
```

禁止：

```text
Healthcare/
Legal/
Product/
Finance/
Education/
```

除非用户明确要求按行业建独立 Vault，否则不要这样建。

## 8. 索引文件规则

默认创建：

```text
00-Home.md
50-MOCs/
```

可以创建 6 个工作模式 MOC：

```text
50-MOCs/MOC - 研究分析.md
50-MOCs/MOC - 项目交付.md
50-MOCs/MOC - 客户个案.md
50-MOCs/MOC - 内容生产.md
50-MOCs/MOC - 流程运营.md
50-MOCs/MOC - 学习成长.md
```

可以做：

- `00-Home.md` 作为总入口。
- 六大工作模式 MOC 作为导航入口。
- 常用行业页作为映射入口。

不可以做：

- 不要给所有行业预建索引。
- 不要给所有可能主题预建索引。
- 不要把 MOC 当作实际内容存放目录。

## 9. 文件命名规则

推荐：

```text
YYYY-MM-DD - 标题.md
```

可加短前缀：

```text
2026-06-30 - product - 用户访谈.md
2026-06-30 - ops - 事故复盘.md
2026-06-30 - legal - 案件时间线.md
```

不要使用：

```text
最终版.md
新建文档.md
未命名.md
杂项.md
```

## 11. 最终执行规则

执行时按顺序：

1. 判断是否能归类。
2. 不能归类，放入 `00-Inbox/`。
3. 能归类，先判断知识层。
4. 再判断工作模式。
5. 如果目标二级目录不存在，按需创建。
6. 如果涉及行业，只创建或更新 `60-Domains/<domain>.md`。
7. 不创建行业一级目录。
8. 不批量创建 24 个二级目录。
9. 不生成具体内容模板。

最终路径格式：

```text
<知识层目录>/<中文工作模式>/<文件名>.md
```

示范：

```text
10-Inputs/研究分析/2026-06-30 - product - 用户访谈.md
20-Process/项目交付/2026-06-30 - PRD 草稿.md
30-Outputs/内容生产/2026-06-30 - 已发布文章.md
40-Feedback/流程运营/2026-06-30 - 事故复盘.md
```

## 12. 什么时候 · 怎么做（5W2H）

§1–11 讲清了「**放哪 / 怎么判**」（What / Where）。这一节反向补上「**什么时候做、谁做、怎么做、做到什么程度**」，让规则可执行。

| 维度 | 内容 |
|---|---|
| **Why 为什么** | 让知识可检索、可复用，画像能被 agent 加载「像你一样做事」；避免 `未命名/杂项` 满天飞、信息散落找不回。 |
| **What 做什么** | 把每条**有长期价值**的新信息**判位 + 写成规范笔记**（层 + 模式 + frontmatter），并维护画像与索引。 |
| **Who 谁做** | 默认 **agent 自动做**（建库 / 在场提炼 / 维护时）；用户只需偶尔把速记丢进 `00-Inbox/` 或说「整理收件箱」。 |
| **Where 放哪** | `<知识层>/<中文工作模式>/`；判不出 → `00-Inbox/`；行业 → `60-Domains/<domain>.md` 映射页（见 §1–7）。 |
| **When 什么时候** | ① 建库时建一级骨架 + 6 张 MOC；② 对话中得到**新**画像 / 知识（库里没有、有长期价值）→ **当场写**；③ 用户喂 URL / 整理收件箱时；④ 某行业 / 某模式**首次**出现 → 才建对应 domain 页 / 二级目录；⑤ 每次入库后 → 更新 `00-Home` + 对应 MOC。 |
| **How 怎么做** | 按 §11 顺序：判能否归类 → 不能进 `00-Inbox/` → 判知识层 → 判工作模式 → 目录不存在按需建 → 写规范笔记（**先去重 / 同标题合并**）→ 涉及行业只更 `60-Domains/<domain>.md` → 更新索引。 |
| **How much 到什么程度** | 只收**有长期价值**的；明文密钥 / token **脱敏不写**；目录**按需建、绝不一次批量 24 个**；同标题**合并、不堆叠重复段落**；画像变更日志只留最近 50 条。 |

### 触发速查（When → How）

| 触发 | 怎么做 |
|---|---|
| 初始化建库 | 建一级目录 + `00-Home` + 6 张 MOC + 空画像；二级 / 行业页**不**预建 |
| 对话得到新信息 | 判位 → 写 `<层>/<模式>/<YYYY-MM-DD> - 标题.md`（去重合并）→ 更新 `00-Home`/MOC |
| 收录链接 / 喂 URL | 抓正文 → 提炼 → 同上落位 |
| 整理收件箱 | 读 `00-Inbox/*.md` → 判位归档 → 原速记移 `00-Inbox/_archived/` |
| 出现新行业 | 只建 / 更 `60-Domains/<domain>.md` 映射页，**不**建行业一级目录 |
| 判不出归属 | 放 `00-Inbox/`，别硬塞，稍后再归 |
